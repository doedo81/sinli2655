# 🏗️ 다중 학급·다중 교사 아키텍처 설계 (v0.1)

> 아침칠판 + 성장 시스템을 **여러 교사가 각자 학급을 만들어 쓰는 구조**로 개편.
> 담임/전담 역할 구분, 성장 데이터 반별 연동, **궁극적으로 공개 서비스**(단계적 출시).
> 핵심 원칙: **개인정보 최소화**(기본 번호/닉네임), 멤버만 접근(보안 규칙).

---

## 1. 현재 구조의 문제

| 항목 | 지금 | 문제 |
|---|---|---|
| 학급 구분 | `rooms/<방이름>` (사람이 직접 입력) | 이름 겹치면 **데이터 섞임**(다들 "5학년1반") |
| 권한 | 없음 | 방 이름만 알면 **아무나** 읽고 씀 |
| 역할 | 없음 | 담임/전담 구분 불가 |
| 보안 규칙 | (확인 필요, 아마 개방) | 남의 반 학생정보 노출 위험 |

**하지만 뼈대는 있음:** 구글 로그인 존재 + 데이터가 이미 반별로 분리(`rooms/<방>/...`, 성장은 `rooms/<방>/rpg`). → 전면 재작성이 아니라 **신원·권한 층을 밑에 까는 작업.**

---

## 2. 목표 데이터 모델

```
/users/<uid>/
    profile: { name, email }
    classes: { <classId>: "owner" | "subject" }   # 내가 속한 학급 + 역할

/classes/<classId>/
    meta:    { name, school, ownerUid, code, createdAt }
    members: { <uid>: { role:"owner"|"subject", name } }
    data/                                          # 기존 rooms/<방> 내용이 여기로
        settings/ dashboard/ rpg/ seating/ ...

/classCodes/<code>: <classId>                      # 코드로 학급 찾기 (예: 5A-7Q2)
```

- `classId` = 자동 생성 고유 키(겹치지 않음). 사람이 쓰는 건 `code`(참여용).
- 성장 데이터(`data/rpg`)는 학급 안에 있어 **담임·전담이 함께** 접근.

---

## 3. 역할·권한

| 역할 | 누구 | 할 수 있는 것 |
|---|---|---|
| 👩‍🏫 **owner (담임)** | 학급 개설자 | 전부 (아침칠판·성장·자리배치·설정·멤버관리) |
| 🧑‍🔬 **subject (전담)** | 코드로 참여 | 성장 점수 주기 + 명단 보기 (설정·칠판 편집 X) |

> 나중에 `viewer`(학부모 열람) 같은 역할도 추가 가능.

---

## 4. 학급 개설·참여 흐름

- **개설(담임)**: [학급 만들기] → classId·code 생성, 본인을 owner로 등록 → 코드 표시
- **참여(전담)**: [코드로 참여] → `classCodes/<code>`로 classId 찾기 → members에 subject로 등록
- **로그인 후 첫 화면**: "내 학급" 목록(개설/참여한 반) → 선택해서 입장
- 지금의 "방 이름 직접 입력"을 이 화면으로 대체

---

## 5. 보안 규칙 (요지, Firebase RTDB)

- `users/<uid>`: 본인만 읽기/쓰기
- `classes/<cid>`: **멤버(members에 내 uid 존재)만** 읽기/쓰기
- 일부(meta·설정·멤버관리)는 **owner만** 쓰기
- `classCodes`: 로그인 사용자가 코드로 조회 가능(참여용), 무분별 열람 방지 설계
- 참여 시 "본인 멤버 노드만 추가 가능" 규칙 (가입 chicken-egg 처리)

> ⚠️ 보안 규칙은 공개 서비스의 **핵심**. 이거 없이는 절대 공개 금지.

---

## 6. 개인정보 최소화 원칙 ⭐ (공개 서비스 부담을 줄이는 열쇠)

- **기본은 번호/닉네임** (예: "3번", "불꽃전사") → 법적으로 개인정보 아님 → 부담 대폭 감소
- 실명 사용은 **담임의 선택 + 책임**으로, 기본 꺼둠
- 생일·사진 등 민감정보는 **수집 최소화 / 선택**
- 데이터는 학급 멤버에게만, 외부 공유 없음

## 8. 안전한 작업 방식

- **새 git 브랜치**에서 개편 → 로컬·소수 베타로 검증 → 확인 후 `main` 병합
- 기존 '신리' 반은 **계속 동작**하게 (이전 스크립트 or 하위호환)
- 큰 변경이므로 각 단계마다 검증·백업
